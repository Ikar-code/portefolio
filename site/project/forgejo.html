Projet de Stage · DevOps & SysAdmin

Installation, Validation et Audit de Sécurité Forgejo

Déploiement d'une solution d'hébergement Git souveraine sur VM Ubuntu

📁 Présentation du Projet

Ce projet réalisé en entreprise durant mon stage a pour objectif l'installation et la mise en service complète de Forgejo sur une machine virtuelle Ubuntu afin de doter la structure d'une plateforme de gestion de dépôts Git auto-hébergée.

Forgejo s'impose comme une alternative libre et légère à GitHub, permettant un contrôle total sur le code source, le suivi des tickets (Issues), les Pull Requests et les dynamiques collaboratives entre développeurs.

⚙️ Déploiement & État du Service

Le processus d'installation a couvert la mise en place des dépendances, le déploiement de l'exécutable et l'enregistrement du démon système.

Le statut du service système a été vérifié et validé :

• Service : active (running)
• Démarrage automatique : enabled

Durcissement de la configuration (Modifications de app.ini)

Pour des raisons évidentes de confidentialité d'infrastructure, des modifications ciblées ont été apportées :

[service]
REQUIRE_SIGNIN_VIEW = true      # Force l'authentification pour lire le contenu
DISABLE_REGISTRATION = true     # Désactive la création autonome de compte
SHOW_REGISTRATION_BUTTON = false # Masque le bouton d'inscription

[server]
LANDING_PAGE = login            # Redirection automatique des anonymes vers la mire

Effet immédiat : La surface d'attaque est réduite, rendant l'inscription fermée et gérée uniquement par l'administrateur.

📧 Serveur SMTP Local & Notifications

Afin de propager les notifications de la forge logicielle vers les collaborateurs, un relais mail a été interconnecté localement.

Étapes : Installation de Postfix en mode "Internet Site" et liaison avec Forgejo via le bloc suivant :

[mailer]
ENABLED = true
PROTOCOL = smtp
HOST = 192.168.194.51
FROM = "Forgejo" <forgejo@localhost>

L'analyse des journaux applicatifs dans /var/log/mail.log confirme le bon traitement et le routage des notifications.

🔐 Audit de Sécurité

Vérification des secrets

L'analyse approfondie du fichier app.ini confirme que les chaînes sensibles (SECRET_KEY, INTERNAL_TOKEN, LFS_JWT_SECRET) sont générées dynamiquement avec une entropie solide et ne sont pas stockées en clair.

Audit des permissions du système de fichiers

⚠️ Vulnérabilité Détectée (Permissions)
Les répertoires applicatifs sous /var/lib/forgejo (notamment data et log) disposaient initialement de droits en drwxr-xr-x, ouvrant l'accès en lecture (o+r) à la base de données SQLite et aux logs à d'autres utilisateurs du système.

Correction appliquée : Application immédiate du principe de moindre privilège en restreignant l'accès exclusif à l'utilisateur système forgejo et au groupe de sécurité de l'infrastructure.

⚔️ Analyse : Forgejo vs GitHub

Critère Forgejo GitHub
Hébergement Auto-hébergé (Interne) Cloud (SaaS)
Souveraineté des données Totale (Contrôle absolu) Limitée (Hébergeur tiers)
Coût d'exploitation Gratuit & Open-Source Modèle commercial / Licences
Interface Épurée, focalisée sur l'essentiel Barres latérales denses, écosystème lourd

Analyse globale : Forgejo offre la légèreté et la gouvernance idéale pour des équipes voulant conserver la maîtrise de leur code, là où GitHub apporte un catalogue d'outils annexes au prix de la dépendance à un acteur tiers.

📌 Conclusion

L'installation sur l'infrastructure virtuelle s'est soldée par un succès opérationnel. L'audit a apporté une plus-value concrète en identifiant un défaut de permissions de fichiers, résolu immédiatement avant la mise en production. La solution est prête et autonome.
